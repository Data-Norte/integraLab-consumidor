IntegraLab Consumidor

Guia para subir, testar e validar o consumidor localmente

Este consumidor recebe webhooks assinados do integraLab-api, solicita as pendencias pela rota de integracao, processa os exames e envia o resultado de volta para a API. O projeto ja esta preparado para o ambiente local seedado.

Arquivos principais

Inicializacao local

Use iniciar-consumidor-teste.cmd para subir o projeto em modo de teste no Windows.

Configuracao

O ambiente local fica em .env. O modelo versionado fica em .env.example.

Servidor

O ponto de entrada da aplicacao e src/server.ts com launcher em src/server.launch.mjs.

Pre-requisitos

  • Node.js instalado.
  • Dependencias do projeto instaladas com npm install.
  • integraLab-api em execucao na porta configurada no .env do consumidor.
  • Bancos de dados localmente acessiveis, caso o fluxo da API precise consultar pendencias reais.
O launcher de teste valida a existencia do arquivo .env antes de iniciar o servidor.

Como iniciar

Opcao recomendada no Windows

iniciar-consumidor-teste.cmd

Opcao por linha de comando

npm run start:test

Build e start de producao local

npm run build
npm start

Endpoints disponiveis

Healthcheck 
GET /health
Health do modulo 
GET /api/lab-apoio/v1/consumer/health
Webhook da API 
POST /api/lab-apoio/v1/consumer/webhook
Disparo manual 
POST /api/lab-apoio/v1/consumer/processar-pendentes
Painel publico QA 
GET /qa
Resumo QA 
GET /api/lab-apoio/v1/consumer/qa/overview
Detalhe do item QA 
GET /api/lab-apoio/v1/consumer/qa/itens/:itemId
Arquivos do item QA 
GET /api/lab-apoio/v1/consumer/qa/itens/:itemId/arquivos/:kind
Arquivos da execucao QA 
GET /api/lab-apoio/v1/consumer/qa/runs/:runId/arquivos/:kind
Limpeza do storage QA 
DELETE /api/lab-apoio/v1/consumer/qa/storage
Endpoints da API QA: :kind aceita entrada, saida, resultado, resposta e pdf para itens, e contexto, webhook e resumo para execucoes.

Fluxo de teste recomendado

  1. Suba o integraLab-api.
  2. Suba o consumidor com iniciar-consumidor-teste.cmd ou npm run start:test.
  3. Valide o healthcheck do consumidor.
  4. Dispare o processamento manual para consumir as pendencias existentes.
  5. Opcionalmente valide o webhook automatico apontando o vinculo da API para a URL do consumidor.

Smoke tests em PowerShell

1. Verificar se o consumidor esta online

Invoke-RestMethod -Method Get `
  -Uri 'http://localhost:3001/health'

2. Disparar processamento manual das pendencias

Invoke-RestMethod -Method Post `
  -Uri 'http://localhost:3001/api/lab-apoio/v1/consumer/processar-pendentes' `
  -ContentType 'application/json' `
  -Body '{}'

3. Consultar o health do modulo do consumidor

Invoke-RestMethod -Method Get `
  -Uri 'http://localhost:3001/api/lab-apoio/v1/consumer/health'

Variaveis importantes

  • INTEGRALAB_API_BASE_URL: URL base da API.
  • LAB_APOIO_TENANT_ID: tenant usado para busca de pendencias.
  • LAB_APOIO_VINCULO_ID: vinculo de integracao da conta de laboratorio.
  • LAB_APOIO_AUTH_SECRET: segredo usado para emitir o JWT de integracao.
  • LAB_APOIO_WEBHOOK_SECRET: segredo HMAC para validar o webhook.
  • LAB_APOIO_SEND_INLINE_PDF: quando false, envia so resultado estruturado.
  • AUTO_PROCESS_WEBHOOK: quando true, processa pendencias ao receber webhook de exames disponiveis.
  • QA_STORAGE_DIR: diretorio local onde o SQLite e os arquivos por tenant ficam salvos.

Comandos uteis

npm test
npm run build
npm run dev
npm run start:test

Observacoes operacionais

  • O consumidor local esta preparado para processar as pendencias seedadas no ambiente atual.
  • Por padrao, o envio de PDF inline esta desabilitado no ambiente local para evitar dependencia de parametros adicionais do Macro.
  • Se voce reexecutar o seed da API, o authSecret e o webhookSecret podem ser rotacionados. Nesse caso o .env do consumidor precisa ser atualizado.
  • A tela /qa fica publica para QA e permite limpar os arquivos gerados localmente.
